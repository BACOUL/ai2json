webpage.ai.v0.1 · Stable JSON Contract

Specification: webpage.ai.v0.1

This document defines the JSON contract returned by A2J when transforming a public webpage. The goal is a predictable schema that AI systems can consume reliably: structured sections, minimal metadata, and a content hash that fingerprints what was read.

Overview

Design goals

webpage.ai.v0.1 is intentionally minimal and stable.

Goals - One URL in → one JSON document out - Stable structure for agents and RAG - Extracted, cleaned main content (not full DOM) - Sections[] preserve reading order and headings - content_hash fingerprints extracted text - Stateless: no storage is required by the spec
Schema

Top-level JSON shape

Required fields are mandatory for compliance with v0.1.

Field Type Required Description
spec string Yes Must be "webpage.ai.v0.1".
url string Yes Requested URL provided by the client.
canonical_url string Yes Canonical URL if detected, otherwise equals url.
last_fetched string Yes UTC timestamp in ISO 8601 format.
language string Yes Best-effort BCP-47-like tag (e.g. en, fr).
content_hash string Yes Hash of the extracted text: sha256:<hex>.
title string Yes Best-effort main title.
summary string Yes Short factual summary of the extracted content.
type string Yes article | homepage | unknown.
sections array Yes Ordered content blocks (see below).
links array No Optional extracted links (implementation-defined).
meta object No Optional best-effort metadata (site, author, dates, tags...).
proof object No Reserved for future integrity proofs.
Sections

sections[] format

Each section represents a contiguous block of content under a heading.

Section object (v0.1) { "heading": "Introduction", "level": 1, "id": "introduction", "text": "Clean extracted text..." }
Field Type Required Description
heading string Yes Best-effort heading label for this block.
level number Yes Heading depth (1 for H1-like, 2 for H2-like, etc.).
id string Yes Stable identifier derived from the heading (slug).
text string Yes Clean text content for that section.
Hashing

content_hash rules

In v0.1, content_hash is computed over the extracted text only (not raw HTML), after normalization. Implementations may differ in extraction heuristics, but should keep hashing deterministic within the same implementation.

Format content_hash = "sha256:" + HEX(SHA-256(extracted_text_normalized_utf8)) Notes - UTF-8 encoding - extracted_text should be the concatenation of section texts in order - normalization should be consistent (whitespace normalized)
