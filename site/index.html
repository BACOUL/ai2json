How A2J works

A thin, opinionated layer between the web and your AI stack.

A2J is intentionally minimal: one endpoint, one spec, stateless by design. It focuses on turning messy HTML into the clean JSON you wish the web had been built with.

1. Fetch and detect

A2J fetches the target URL, follows redirects when needed and detects the main content using a reader-style heuristic instead of scraping everything.

2. Clean and normalize

Navigation, ads, scripts and layout noise are stripped away. The remaining content is normalized with consistent spacing and line-breaking.

3. Structure into sections

Headings and paragraphs are grouped into sections[], each with a heading, level, id and text field, so agents and RAG pipelines can reason section by section.

4. Summarize and fingerprint

A factual summary is generated and a SHA-256 hash over the extracted text becomes a stable fingerprint. You know exactly what was read and when.

JSON format

The webpage.ai.v0.1 contract.

Every A2J response is a single JSON document, designed to be easy to ingest and stable over time. No guessing, no reverse-engineering, no undocumented fields.

Field Type Purpose
spec string Always "webpage.ai.v0.1" for this version.
url string URL requested by the client.
canonical_url string Canonical URL if detected, otherwise mirrors url.
last_fetched string ISO 8601 UTC timestamp of the snapshot.
language string Best-guess language, for example "en" or "fr".
content_hash string SHA-256 of extracted text, prefixed with sha256:.
title string Main page title.
summary string Short factual summary of the page content.
type string One of "article", "homepage", or "unknown".
sections[] array Ordered blocks with heading, level, id, text.
links[] array Optional extracted links for navigation or reference.
meta object Best-effort metadata such as site name, author, dates, tags.
proof object Reserved for future content-integrity proofs.

The formal specification lives in spec/webpage-ai-v0.1.md in the repository and is kept stable for all of v0.1.

API at a glance

A single stateless endpoint in v0.1.

The first version of A2J is intentionally small: one endpoint, one spec, no surprises. You send a URL and an API key, you receive a webpage.ai.v0.1 document.

Base URL (example) https://api.ai2json.com Endpoint GET /v1/transform?url=<ENCODED_URL> Headers Authorization: Bearer YOUR_API_KEY Accept: application/json Query parameters url Required. Public page URL to transform. lang Optional. Preferred language for future advanced summaries. no_cache Optional. If true, forces a fresh fetch instead of using cache. Behaviour 200 OK Body is a webpage.ai.v0.1 JSON document. 400 Invalid or missing url parameter. 401 Missing or invalid API key. 4xx Other client errors. 5xx Upstream or internal errors, with JSON error payload.

Full documentation, examples and language-specific snippets live in the GitHub repository and will grow with the product.

Roadmap

From focused utility to a standard way for AI to read the web.

A2J is built to stay small at the surface and powerful under the hood. The goal is to keep the core contract stable while adding capabilities that matter for real AI workloads.

v0.1 · HTML → JSON

Single endpoint, webpage.ai.v0.1 spec, basic extraction and summary, single early-access API key flow.

v0.2 · Smarter summaries

LLM-powered summaries with explicit language control, improved metadata, and better handling of long technical documents.

v0.3 · Dynamic pages

Optional JavaScript rendering for SPAs and dashboards, with the same JSON contract for downstream consumers.

v1.0 · Production standard

Dashboards, key management, quotas, monitoring, webhooks, and a formally versioned specification for long-term stability.
